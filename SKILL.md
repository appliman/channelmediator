# ChannelMediator Implementation Skill

## Purpose

Use this skill when an LLM coding agent such as Codex, Claude, or GitHub Copilot must add, configure, or extend ChannelMediator in a .NET solution. It covers in-process mediator usage, Minimal API and gRPC source generators, generated HTTP/gRPC clients, and distributed Pub/Sub or queue processing with Azure Service Bus and RabbitMQ.

## When to Use

Use this skill for tasks that mention any of the following:

- ChannelMediator setup, migration from MediatR, or mediator registration.
- `IRequest<TResponse>`, `IRequest`, `INotification`, stream requests, handlers, or pipeline behaviors.
- Source-generated Minimal APIs, HTTP clients, gRPC services, or gRPC clients.
- `ChannelMediator.AzureBus` or Azure Service Bus topics, queues, subscriptions, readers, publishers, or Pub/Sub.
- `ChannelMediator.RabbitMQ` or RabbitMQ exchanges, queues, bindings, readers, publishers, or Pub/Sub.
- Blazor, console, worker, API, or microservice projects that communicate through ChannelMediator.

## Repository Context

ChannelMediator is a high-performance mediator library built on `System.Threading.Channels` with a MediatR-compatible API.

Main projects:

| Project | Responsibility |
| --- | --- |
| `ChannelMediator.Contracts` | Public contracts such as `IRequest<T>`, `IRequest`, `INotification`, `Unit`, stream contracts. Keep dependency-free. |
| `ChannelMediator` | Core mediator, DI registration, handler dispatch, notifications, streaming, and pipeline behaviors. |
| `ChannelMediator.AzureBus` | Azure Service Bus integration for distributed notifications and queued commands. |
| `ChannelMediator.RabbitMQ` | RabbitMQ integration for distributed notifications and queued commands. |
| `ChannelMediator.ApiGenerators.Abstraction` | Shared generator attributes. |
| `ChannelMediator.MinimalApiGenerator` | Source generator for ASP.NET Core Minimal API endpoints. |
| `ChannelMediator.ApiClientGenerator` | Source generator for `HttpClient`-based request handlers. |
| `ChannelMediator.GrpcGenerator` | Source generator for code-first gRPC services. |
| `ChannelMediator.GrpcClientGenerator` | Source generator for gRPC request handlers. |
| `ChannelMediator.Tests` | xUnit tests. |
| `Samples/*` | Console, Blazor, Minimal API, gRPC, Azure Service Bus, and RabbitMQ samples. |

Targets are currently .NET 8, .NET 9, .NET 10, and .NET Standard 2.0 where applicable. Do not change target frameworks, SDK versions, or language version unless the user explicitly asks.

## Core Rules for LLM Agents

- Follow the existing project conventions before introducing new patterns.
- Prefer small, focused changes.
- Do not edit generated files (`*.g.cs`, files marked `<auto-generated>`, or generated source output).
- Public APIs require XML documentation.
- Use nullable-safe code and early guards such as `ArgumentNullException.ThrowIfNull`.
- Pass `CancellationToken` through every async path.
- Do not use `.Result`, `.Wait()`, sync-over-async, or fire-and-forget in new core code.
- In this repository, do not add `ConfigureAwait(false)`; use plain `await`.
- Avoid runtime reflection in hot paths. Prefer generics, source generation, or compile-time patterns.
- Do not add abstractions unless they are required for external dependencies, testing, or an existing extension point.
- For tests, use the existing xUnit style with behavior-focused test names.

## In-Process Mediator Setup

Register ChannelMediator in the application service collection and include the assemblies that contain handlers:

```csharp
using ChannelMediator;
using Microsoft.Extensions.DependencyInjection;

services.AddChannelMediator(
	config => config.Strategy = NotificationPublishStrategy.Parallel,
	typeof(Program).Assembly);
```

When several handlers exist for the same request type, the first registered handler wins. If passing multiple assemblies, put the assembly containing the preferred default handler first.

Use these contracts:

```csharp
public record GetProductRequest(int Id) : IRequest<Product?>;

public sealed class GetProductHandler : IRequestHandler<GetProductRequest, Product?>
{
	public Task<Product?> Handle(GetProductRequest request, CancellationToken cancellationToken)
	{
		return Task.FromResult<Product?>(new Product(request.Id, "Widget"));
	}
}

public record RebuildCatalogRequest : IRequest;

public sealed class RebuildCatalogHandler : IRequestHandler<RebuildCatalogRequest>
{
	public Task Handle(RebuildCatalogRequest request, CancellationToken cancellationToken)
	{
		return Task.CompletedTask;
	}
}

public record ProductAddedNotification(int ProductId) : INotification;

public sealed class ProductAddedHandler : INotificationHandler<ProductAddedNotification>
{
	public Task Handle(ProductAddedNotification notification, CancellationToken cancellationToken)
	{
		return Task.CompletedTask;
	}
}
```

Use the mediator through `IMediator`:

```csharp
var product = await mediator.Send(new GetProductRequest(42), cancellationToken);
await mediator.Send(new RebuildCatalogRequest(), cancellationToken);
await mediator.Publish(new ProductAddedNotification(42), cancellationToken);
```

## Migrating from MediatR to ChannelMediator

ChannelMediator is designed to be close to a drop-in replacement for MediatR. Most request, command, notification, handler, and `Send` / `Publish` call-site code can remain structurally identical.

### Migration Checklist

1. Replace the MediatR package reference with ChannelMediator references.
2. Replace `using MediatR;` with `using ChannelMediator;`.
3. Replace `services.AddMediatR(...)` with `services.AddChannelMediator(...)`.
4. Update pipeline behaviors from MediatR's `Handle` / `Task<TResponse>` shape to ChannelMediator's `HandleAsync` / `ValueTask<TResponse>` shape.
5. Replace open-generic pipeline registrations with `AddOpenPipelineBehavior(...)`.
6. Replace `ISender` and `IPublisher` injection with `IMediator` injection.
7. Build and fix any remaining namespace or behavior-registration errors.

### Package and Registration Migration

For application projects, replace the MediatR package with `ChannelMediator`. Inside this repository, use the local project reference instead.

```xml
<!-- Before -->
<PackageReference Include="MediatR" Version="12.*" />

<!-- After, consuming application -->
<PackageReference Include="ChannelMediator" Version="x.y.z" />

<!-- After, inside this repository -->
<ProjectReference Include="..\ChannelMediator\ChannelMediator.csproj" />
```

Replace registration:

```csharp
// Before: MediatR
services.AddMediatR(config =>
{
	config.RegisterServicesFromAssembly(typeof(Program).Assembly);
});

// After: ChannelMediator
services.AddChannelMediator(
	config => config.Strategy = NotificationPublishStrategy.Sequential,
	typeof(Program).Assembly);
```

Use `NotificationPublishStrategy.Sequential` when preserving MediatR-like notification ordering. Use `NotificationPublishStrategy.Parallel` only when handlers are independent and safe to run concurrently.

### Interface and Type Mapping

| MediatR | ChannelMediator | Migration note |
| --- | --- | --- |
| `MediatR.IRequest<TResponse>` | `ChannelMediator.IRequest<TResponse>` | Change namespace only. |
| `MediatR.IRequest` | `ChannelMediator.IRequest` | Change namespace only. |
| `MediatR.INotification` | `ChannelMediator.INotification` | Change namespace only. |
| `MediatR.IRequestHandler<TRequest, TResponse>` | `ChannelMediator.IRequestHandler<TRequest, TResponse>` | Handler method stays `Handle`. |
| `MediatR.IRequestHandler<TRequest>` | `ChannelMediator.IRequestHandler<TRequest>` | Handler method stays `Handle`. |
| `MediatR.INotificationHandler<TNotification>` | `ChannelMediator.INotificationHandler<TNotification>` | Handler method stays `Handle`. |
| `MediatR.Unit` | `ChannelMediator.Unit` | Change namespace only. |
| `ISender`, `IPublisher` | `IMediator` | Inject `IMediator` for both `Send` and `Publish`. |

Request, command, notification, and handler implementations usually only require the namespace change:

```csharp
using ChannelMediator;

public record GetOrderQuery(int OrderId) : IRequest<OrderDto>;

public sealed class GetOrderHandler : IRequestHandler<GetOrderQuery, OrderDto>
{
	public Task<OrderDto> Handle(GetOrderQuery request, CancellationToken cancellationToken)
	{
		return Task.FromResult(new OrderDto(request.OrderId));
	}
}
```

### Pipeline Behavior Migration

MediatR pipeline behaviors use `Handle` and `Task<TResponse>`. ChannelMediator pipeline behaviors use `HandleAsync` and `ValueTask<TResponse>`.

```csharp
// Before: MediatR
public sealed class LoggingBehavior<TRequest, TResponse>
	: IPipelineBehavior<TRequest, TResponse>
	where TRequest : IRequest<TResponse>
{
	public async Task<TResponse> Handle(
		TRequest request,
		RequestHandlerDelegate<TResponse> next,
		CancellationToken cancellationToken)
	{
		return await next();
	}
}

// After: ChannelMediator global behavior
public sealed class LoggingBehavior<TRequest, TResponse>
	: IPipelineBehavior<TRequest, TResponse>, IPipelineBehavior
	where TRequest : IRequest<TResponse>
{
	public async ValueTask<TResponse> HandleAsync(
		TRequest request,
		RequestHandlerDelegate<TResponse> next,
		CancellationToken cancellationToken)
	{
		return await next();
	}
}
```

Register global behaviors explicitly:

```csharp
// Before: MediatR
services.AddScoped(typeof(IPipelineBehavior<,>), typeof(LoggingBehavior<,>));

// After: ChannelMediator
services.AddOpenPipelineBehavior(typeof(LoggingBehavior<,>));
```

For a behavior that must apply to one request type only, prefer the specific registration API:

```csharp
services.AddPipelineBehavior<GetOrderQuery, OrderDto, ValidateGetOrderBehavior<GetOrderQuery, OrderDto>>();
```

### Streaming Migration

Streaming request and handler shapes remain the same. Keep `IStreamRequest<TResponse>`, `IStreamRequestHandler<TRequest, TResponse>`, and `mediator.CreateStream(...)`, then change the namespace to `ChannelMediator`.

If stream pipeline behaviors exist, verify their signatures against `IStreamPipelineBehavior<TRequest, TResponse>` and keep cancellation flowing through `WithCancellation(cancellationToken)` when enumerating nested streams.

### Post-Migration Checks

- Search for remaining `using MediatR;`, `AddMediatR`, `ISender`, and `IPublisher` references.
- Verify open-generic behaviors implement the non-generic `IPipelineBehavior` marker when they are intended to be global.
- Verify notification strategy: use sequential for behavior parity, parallel for throughput when safe.
- Confirm the assemblies passed to `AddChannelMediator` contain all handlers.
- Run the relevant tests and at least a build of the migrated application.

## Pipeline Behaviors

Use pipeline behaviors for cross-cutting request concerns such as validation, tracing, metrics, authorization, and transactions.

Global behavior registration:

```csharp
services.AddOpenPipelineBehavior(typeof(LoggingBehavior<,>));
```

Specific behavior registration:

```csharp
services.AddPipelineBehavior<GetProductRequest, Product?, ValidateProductRequestBehavior<GetProductRequest, Product?>>();
```

Do not put notification fan-out logic in request pipeline behaviors. Use `INotification` handlers or distributed Pub/Sub integrations instead.

## Streaming Requests

Use `IStreamRequest<T>` and `IStreamRequestHandler<TRequest, TResponse>` for incremental result sets. Streaming bypasses the channel pump and should stream data end-to-end.

```csharp
public record GetOrderLinesQuery(int OrderId) : IStreamRequest<OrderLineDto>;

public sealed class GetOrderLinesHandler : IStreamRequestHandler<GetOrderLinesQuery, OrderLineDto>
{
	public async IAsyncEnumerable<OrderLineDto> Handle(
		GetOrderLinesQuery request,
		[EnumeratorCancellation] CancellationToken cancellationToken)
	{
		await foreach (var line in LoadLinesAsync(request.OrderId, cancellationToken))
		{
			yield return line;
		}
	}
}
```

## Minimal API Generator Setup

Use the generator when request contracts should become ASP.NET Core Minimal API endpoints.

### Contracts Project

Reference:

- `ChannelMediator.Contracts`
- `ChannelMediator.ApiGenerators.Abstraction`

Define request records and annotate them with `[EndpointApi]`:

```csharp
using ChannelMediator;
using ChannelMediator.ApiGenerators.Abstraction;

[EndpointApi(
	GroupName = "Catalog",
	Path = "products",
	UseHttpStandardVerbs = true,
	Summary = "Gets a product by id",
	Tags = new[] { "Products" })]
public record GetProductRequest(int Id) : IRequest<Product?>;
```

Important `[EndpointApi]` options:

- `GroupName`: maps to `/api/{groupName}`.
- `Path`: path segment after the group.
- `UseHttpStandardVerbs`: infers HTTP verb from request name.
- `Protocol`: `Http`, `Grpc`, or `Both`.
- `AuthenticationSchemes`: emits authorization requirements.
- `Summary`, `Description`, `Tags`: OpenAPI metadata.

HTTP verb inference when `UseHttpStandardVerbs = true`:

| Request prefix | HTTP verb | Binding |
| --- | --- | --- |
| `Get*` | GET | Query string |
| `Delete*` | DELETE | Query string |
| `Put*`, `Update*` | PUT | JSON body |
| `Post*`, `Create*`, `Save*` | POST | JSON body |

If `UseHttpStandardVerbs = false`, generated endpoints use POST with a JSON body.

### Server Project

Reference:

- `ChannelMediator`
- Contracts project
- `ChannelMediator.ApiGenerators.Abstraction`
- `ChannelMediator.MinimalApiGenerator` as an analyzer

Create a partial mapper class:

```csharp
using ChannelMediator.ApiGenerators.Abstraction;

[MapApiExtension]
public static partial class CatalogApiMapper;
```

Register services and call the generated mapper:

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddChannelMediator(null, typeof(Program).Assembly);
builder.Services.AddOpenApi();

var app = builder.Build();

app.MapCatalogApiMapper();

await app.RunAsync();
```

Implement the real request handlers in the server project. The generated endpoints call `IMediator.Send` or `IMediator.CreateStream`.

## HTTP API Client Generator Setup

Use this generator when a client application should call generated Minimal APIs by using `mediator.Send(...)`.

Client project references:

- `ChannelMediator`
- Contracts project
- `ChannelMediator.ApiClientGenerator` as an analyzer
- `Microsoft.Extensions.Http`

Add an assembly attribute that points to the contracts assembly:

```csharp
using ChannelMediator.ApiGenerators.Abstraction;
using MyApp.Contracts;

[assembly: ApiClient(typeof(GetProductRequest), HttpClientName = "ApiClient")]
```

Register the mediator and named `HttpClient`:

```csharp
services.AddChannelMediator(null, typeof(Program).Assembly);
services.AddHttpClient("ApiClient")
	.ConfigureHttpClient(client => client.BaseAddress = new Uri("https://localhost:7031/api/"));
```

The generator emits `IRequestHandler<TRequest, TResponse>` implementations. Client code remains mediator-based:

```csharp
var product = await mediator.Send(new GetProductRequest(42), cancellationToken);
```

Ensure `JsonOptions` on `[ApiClient]` matches `JsonOptions` on `[MapApiExtension]`.

## Pub/Sub and Queue Processing: Conceptual Model

ChannelMediator supports two distributed messaging patterns through Azure Service Bus and RabbitMQ:

| Pattern | Use | Local mediator equivalent | Distributed method |
| --- | --- | --- | --- |
| Request queue | Asynchronous command processing with competing consumers | `mediator.Send(command)` | `mediator.EnqueueRequest(command)` |
| Notification Pub/Sub | Fan-out event delivery to every subscriber | `mediator.Publish(notification)` | `mediator.Notify(notification)` |

`EnqueueRequest` requires a command-style `IRequest` with no response. Do not use distributed queue enqueue for `IRequest<TResponse>` queries because there is no immediate response path.

The transport extension methods are named `Notify` and `EnqueueRequest` in both Azure Service Bus and RabbitMQ integrations. Add only the namespace for the transport being used in a file to avoid ambiguous extension methods:

```csharp
using ChannelMediator.AzureBus;
// or
using ChannelMediator.RabbitMQ;
```

## Azure Service Bus Setup

Use `ChannelMediator.AzureBus` for cloud-hosted Pub/Sub and queue processing.

Register the mediator and Azure Service Bus integration:

```csharp
using ChannelMediator;
using ChannelMediator.AzureBus;

services.AddChannelMediator(config =>
{
	config.Strategy = NotificationPublishStrategy.Parallel;

	config.UseChannelMediatorAzureBus(options =>
	{
		options.ConnectionString = configuration.GetConnectionString("AzureBusConnectionString") ?? string.Empty;
		options.Prefix = "myapp";
		options.TopicSubscriberName = "catalog-service";
		options.MaxConcurrentCalls = 4;
		options.AutoCompleteMessages = true;
		options.ProcessMode = AzureServiceBusMode.Live;

		options.AddAzureQueueRequestReader<RebuildCatalogRequest>();
		options.AddAzureBusTopicNotificationReader<ProductAddedNotification>("catalog-service");
	});
}, typeof(Program).Assembly);
```

Publish and enqueue:

```csharp
await mediator.Notify(new ProductAddedNotification(42), cancellationToken);
await mediator.EnqueueRequest(new RebuildCatalogRequest(), cancellationToken);
```

Reader options:

- `AddAzureQueueRequestReader<TRequest>()`: consumes queue messages and dispatches `mediator.Send(request)` locally.
- `AddAzureBusTopicNotificationReader<TNotification>(subscriptionName)`: consumes a topic subscription and dispatches `mediator.Publish(notification)` locally.
- `AddAllAzureBusTopicNotification()`: subscribes to all topics matching the configured prefix. It requires both `Prefix` and `TopicSubscriberName`.

Modes:

- `AzureServiceBusMode.Live`: uses real Azure Service Bus clients and creates entities as needed.
- `AzureServiceBusMode.Mock`: processes in-process, useful for development and tests.

Do not hardcode connection strings. Use configuration, user secrets, environment variables, or managed deployment secrets.

## RabbitMQ Setup

Use `ChannelMediator.RabbitMQ` for self-hosted, local, on-premise, or broker-based Pub/Sub and queue processing.

Register the mediator and RabbitMQ integration:

```csharp
using ChannelMediator;
using ChannelMediator.RabbitMQ;

services.AddChannelMediator(config =>
{
	config.Strategy = NotificationPublishStrategy.Parallel;

	config.UseChannelMediatorRabbitMQ(options =>
	{
		options.Prefix = "myapp";
		options.HostName = "localhost";
		options.Port = 5672;
		options.UserName = "guest";
		options.Password = "guest";
		options.VirtualHost = "/";
		options.TopicSubscriberName = "catalog-service";
		options.PrefetchCount = 8;
		options.ProcessMode = RabbitMqMode.Live;

		options.AddRabbitMqQueueRequestReader<RebuildCatalogRequest>();
		options.AddRabbitMqTopicNotificationReader<ProductAddedNotification>("catalog-service");
	});
}, typeof(Program).Assembly);
```

Publish and enqueue:

```csharp
await mediator.Notify(new ProductAddedNotification(42), cancellationToken);
await mediator.EnqueueRequest(new RebuildCatalogRequest(), cancellationToken);
```

Reader options:

- `AddRabbitMqQueueRequestReader<TRequest>()`: consumes queue messages and dispatches `mediator.Send(request)` locally.
- `AddRabbitMqTopicNotificationReader<TNotification>(subscriptionName)`: consumes exchange subscription messages and dispatches `mediator.Publish(notification)` locally.
- `AddAllRabbitMqTopicNotification()`: subscribes to all exchanges matching the configured prefix. It requires both `Prefix` and `TopicSubscriberName`.

Modes:

- `RabbitMqMode.Live`: uses a real RabbitMQ broker and creates exchanges, queues, and bindings as needed.
- `RabbitMqMode.Mock`: processes in-process, useful for development and tests.

Do not hardcode production broker credentials. Use configuration, user secrets, environment variables, or deployment secrets.

## Blazor Usage Guidance

If the workspace includes Blazor, prefer these patterns:

- Blazor WebAssembly clients should use generated HTTP or gRPC client handlers and call `mediator.Send(...)` from components or services.
- Blazor Server can use in-process handlers for server-side work, or generated clients when calling a separate backend.
- Keep UI components thin. Put mediator calls behind component methods or application services.
- Pass `CancellationToken` when the calling lifecycle provides one.
- Avoid placing transport configuration directly in UI components; configure it in startup/DI.

## Source Generator Troubleshooting

When generated code is missing:

1. Confirm the generator project/package is referenced as an analyzer.
2. Confirm the contracts project references `ChannelMediator.ApiGenerators.Abstraction`.
3. Confirm the request type is annotated with `[EndpointApi]`.
4. Confirm the server mapper is a `partial` class annotated with `[MapApiExtension]` or `[GrpcServiceExtension]`.
5. Confirm the client assembly has `[assembly: ApiClient(...)]` or `[assembly: GrpcClient(...)]`.
6. Rebuild the project and inspect compiler diagnostics before editing generator output.

## Validation Checklist

After changes, run the narrowest relevant validation first, then broader validation if needed:

- Build the changed project.
- Run relevant xUnit tests from `ChannelMediator.Tests`.
- For generator changes, build at least one affected sample project.
- For Pub/Sub changes, run or inspect the Azure Service Bus or RabbitMQ sample paths as appropriate.
- For public API changes, add or update tests and XML documentation.

## Common Mistakes to Avoid

- Referencing source generator projects as normal runtime references instead of analyzer references.
- Adding both `ChannelMediator.AzureBus` and `ChannelMediator.RabbitMQ` namespaces in the same file and causing ambiguous `Notify` or `EnqueueRequest` extension methods.
- Using `IRequest<TResponse>` with distributed `EnqueueRequest`; queued requests must be command-style `IRequest`.
- Forgetting to register the assembly that contains handlers in `AddChannelMediator`.
- Configuring `AddAllAzureBusTopicNotification()` or `AddAllRabbitMqTopicNotification()` without setting both `Prefix` and `TopicSubscriberName`.
- Editing generated files instead of fixing attributes, contracts, mapper declarations, or generator code.
- Hardcoding secrets or connection strings in source code.
- Blocking async code with `.Result` or `.Wait()`.
